    IRSA: Zero-Code Identity with Pod Token Lifecycle

    IAM Roles for Service Accounts (IRSA) lets EKS Pods assume AWS IAM roles without any credentials in application code. The entire identity handshake is handled transparently by Kubernetes and AWS — your app just makes normal SDK calls and the right permissions appear.

    This architectural pattern is called Security Transparency or Zero-Code Identity.

    The Lifecycle of a Pod's Identity Token

    1. Token Request Service

    Every EKS cluster runs a feature called Service Account Token Volume Projection. When you annotate a Kubernetes ServiceAccount with an IAM role ARN:

    apiVersion: v1
kind: ServiceAccount
metadata:
  name: alb-controller
  namespace: kube-system
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::ACCOUNT_ID:role/AWSLoadBalancerControllerRole

    ...the Kubernetes API Server detects that any Pod using this ServiceAccount needs an identity token. It does not wait to be asked — it prepares one automatically.

    2. Automatic Injection (Volume Mount)

    When the Pod starts, Kubernetes injects a token file into the container's filesystem via a projected Volume Mount. No Dockerfile change, no application code change — it simply appears on disk at:

    /var/run/secrets/eks.amazonaws.com/serviceaccount/token

    You can verify this from inside a running Pod:

    kubectl exec -n kube-system <pod-name> -- cat /var/run/secrets/eks.amazonaws.com/serviceaccount/token

    3. What's Inside the Token (JWT)

    The file contains a JSON Web Token (JWT) — a cryptographically signed, base64-encoded claims document. Decode it at jwt.io and you'll see three key claims:

    Claim Field Example value
    Issuer iss https://oidc.eks.us-east-1.amazonaws.com/id/CLUSTER_ID
    Subject sub system:serviceaccount:kube-system:alb-controller
    Audience aud sts.amazonaws.com
    Expiry exp ~1 hour from issue time

    The signature is the critical part: it is a mathematical proof generated by the cluster's private key. AWS can verify it using the cluster's public key, which EKS publishes at the OIDC URL.

    4. Token Consumption by the AWS SDK

    The AWS SDK (baked into your application, e.g. the ALB Ingress Controller) is pre-wired to look for this file. It:

    1. Reads the JWT from disk.
    2. Sends it to AWS STS via an AssumeRoleWithWebIdentity API call.
    3. Receives back short-lived credentials (Access Key + Secret + Session Token, valid ~15 min–1 hr).
    4. Uses those credentials for all subsequent AWS API calls.
    Pod → STS: "Here is my JWT, I want to assume arn:aws:iam::...:role/AWSLoadBalancerControllerRole"
STS → OIDC endpoint: "Is this signature valid?"
OIDC endpoint → STS: "Yes, verified."
STS → Pod: "Here are your temporary credentials."

    The SDK handles token refresh automatically when credentials near expiry.

    Why This Design Is Considered "Genius"

    Zero Application Configuration

    Your application code has no knowledge of IAM, credential rotation, or OIDC. It simply calls boto3.client('s3').list_buckets() or equivalent — the SDK handles everything beneath the surface. This makes application code portable across environments (dev, staging, prod) without modification.

    Enhanced Security — Short-Lived Tokens

    The JWT expires in ~1 hour. If an attacker exfiltrates the token file, they have a shrinking window before it becomes worthless. Compare this to long-lived IAM access keys that remain valid until manually rotated.

    IRSA tokens cannot be used outside the cluster context — the audience claim (aud: sts.amazonaws.com) and subject claim tie the token to a specific ServiceAccount, not a human user.

    Cryptographic Trust — No Passwords

    AWS STS does not trust the Pod because it politely asked for access. It trusts the Pod because the JWT's mathematical signature matches the public key published at the cluster's OIDC discovery URL. Without the cluster's private key — which only EKS holds — no one can forge a valid token.

    Summary

    Role Actor What it does
    Identity Provider EKS cluster Creates and signs the JWT, publishes public key via OIDC
    Bearer Pod Reads the token file and presents it to AWS
    Verifier AWS STS Validates the JWT signature, issues temporary credentials
    Consumer Application (AWS SDK) Uses the temporary credentials for API calls

    Why the Transparency Is Intentional

    In a microservices architecture, you want to decouple environment configuration from application code. The "hidden" nature of IRSA means:

    • Platform/DevOps engineers can update IAM policies or swap roles without touching application code.
    • Developers write business logic without caring about credential management.
    • Security teams can audit permissions centrally via IAM, not scattered across config files.

    The result: security policies are a platform concern, and applications remain clean and portable.
