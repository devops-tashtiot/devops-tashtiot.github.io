🌐 VPC Architecture

Hub-and-Spoke network design with distributed egress and centralized inspection

Overview

Each workload account runs a spoke VPC in il-central-1 (Israel). All outbound internet traffic is inspected by a centralized Network Firewall that lives in a shared hub account — without any Transit Gateway. After inspection, traffic exits through the spoke's own NAT Gateway and Internet Gateway. This is the distributed egress model: shared inspection, local exit.

Traffic Flow — Egress
Your app Spoke subnet — workloads
↓  0.0.0.0/0 → firewall VPC endpoint
Hub firewall Hub account — centralized Network Firewall
↓  approved — traffic returns to firewall subnet
NAT Gateway NAT subnet — per-AZ egress
↓  0.0.0.0/0 → Internet Gateway
Internet Public — traffic leaves your account

Who Owns What
Component Lives in Shared across accounts?
Network Firewall (inspection rules) Hub account ✅ Yes — all spoke VPCs share it
Firewall VPC endpoints Spoke account (firewall subnets) ❌ No — one per spoke VPC
NAT Gateway Spoke account (NAT subnets) ❌ No — one per AZ per spoke
Internet Gateway Spoke account ❌ No — one per spoke VPC
Design Principles
🔍
Centralized inspection
All egress traffic passes through a single shared firewall. Security rules are managed once, enforced everywhere.
🚪
Distributed egress
Each spoke has its own NAT GW and IGW. No Transit Gateway needed — internet exit is local, only inspection is central.
⚡
High availability
Every subnet type is deployed in two AZs. Each AZ has its own firewall endpoint, NAT Gateway, and spoke subnet.
🔒
No direct internet access
No resource has a public IP. All subnets are isolated — nothing reaches the internet without passing the firewall first.
